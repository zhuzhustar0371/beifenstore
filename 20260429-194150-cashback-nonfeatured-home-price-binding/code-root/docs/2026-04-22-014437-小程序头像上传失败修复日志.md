## 一、问题分析

从用户截图和代码链路排查，上传失败发生在 `pages/login/login.js` 的头像上传阶段，而不是昵称保存阶段。

排查结论：

1. 前端流程是：`chooseAvatar` 取头像 -> `wx.uploadFile` 上传到 `/api/auth/profile/avatar`。
2. 后端上传接口原先有一条过严的 MIME 过滤：
   - 仅接受 `contentType.startsWith("image/")`
3. `wx.uploadFile` 在部分机型/环境下上传头像时，`contentType` 可能是：
   - `application/octet-stream`
   - 或者为空
4. 这种情况下后端直接抛 `BusinessException("仅支持图片文件")`，前端只看到统一的“上传头像失败”。

结论：
- 本次失败的主要原因是后端对头像 MIME 类型校验过严，不是登录 token 主流程问题。

## 三、本次代码修改

### 1) 后端放宽头像上传校验

文件：`backend-api/src/main/java/com/zhixi/backend/controller/AuthController.java`

修改点：

- 取消“必须 `contentType.startsWith("image/")`”的硬校验
- 改为：
  - 先校验文件大小
  - 再检查扩展名是否在图片白名单内
  - 如果 MIME 非 image，但扩展名是 `png/jpg/jpeg/gif/webp/bmp/heic/heif`，允许通过

作用：
- 兼容 `wx.uploadFile` 上来的 `application/octet-stream` 或缺失 MIME 情况
- 避免头像文件被误判成非图片

关键位置：
- `AuthController.java:141`

### 2) 前端头像选择补充兼容 fallback

文件：`wechat-app/pages/login/login.js`

修改点：

- `chooseAvatar` 事件拿不到 `avatarUrl` 时，自动 fallback 到：
  - `wx.chooseMedia`
  - 或 `wx.chooseImage`
- 上传接口 `wx.uploadFile` 失败时，优先读取后端返回的真实 `message`
- 增加更明确的本地提示：
  - `选择头像失败`
  - `资料同步失败，已先进入程序`

作用：
- 兼容不同微信版本/开发者工具对 `chooseAvatar` 的行为差异
- 让前端尽量拿到真实错误，而不是只有通用“上传头像失败”

关键位置：
- `login.js:124`
- `login.js:155`
- `login.js:257`
