## 2. 问题现象

- 用户反馈：登录弹窗里展示的二维码，手机扫完之后还会跳到另一个微信二维码页面。
- 直接影响：
  - 当前弹窗里的二维码不是真正可确认登录的微信 PC 登录二维码。
  - 用户需要“先扫一个二维码，再看到另一个二维码”，流程错误。
  - 现有流程下，PC 侧微信登录体验不符合微信开放平台标准扫码登录方式。

## 3. 修改前分析结论

### 3.1 根因定位

检查文件：

- `G:\zhiximini\zhixi-website\frontend\src\components\LoginModal.vue`
- `G:\zhiximini\zhixi-website\frontend\src\api.js`
- `G:\zhiximini\backend-api\src\main\java\com\zhixi\backend\service\UserAuthService.java`

定位结果：

1. 后端 `createWechatQrSession()` 返回的是微信开放平台授权地址 `wechatAuthUrl`，形如：
   - `https://open.weixin.qq.com/connect/qrconnect?...`
2. 前端此前把这个 `wechatAuthUrl` 又用本地二维码库编码成了一张二维码图片。
3. 手机扫描的其实不是“微信 PC 登录二维码”，而只是一个“打开微信登录页的链接二维码”。
4. 所以手机扫码后才会再进入微信官方页面，并出现真正需要确认登录的二维码。

### 3.2 设计判断

本次不改后端 OAuth、状态轮询和用户同步逻辑，原因如下：

1. 后端已经正确生成 `scene/state`、处理微信回调、同步昵称头像并创建系统 token。
2. 真正的问题只在前端二维码渲染方式。
3. 微信开放平台官方内嵌扫码方式应使用 `wxLogin.js`，由微信官方 iframe 在 PC 页面内直接渲染真正的登录二维码。

### 3.3 官方实现核验

为避免误用组件，本次额外核对了官方脚本与文档：

- 官方脚本：`https://res.wx.qq.com/connect/zh_CN/htmledition/js/wxLogin.js`
- 官方文档：`https://developers.weixin.qq.com/doc/oplatform/Website_App/WeChat_Login/Wechat_Login.html`

确认结论：

1. PC 网站内嵌微信登录二维码的标准方式是 `new WxLogin(...)`。
2. `self_redirect=true` 时，手机确认登录后会在 iframe 内跳转到 `redirect_uri`。
3. 这与当前前端已有的 `scene` 轮询模式兼容，因此无需推翻现有后端链路。
