## 一、问题分析

用户反馈“还是无法上传头像”。结合当前代码和参考项目排查后，问题重点不在小程序选择头像本身，而在服务端上传存储链路：

1. 当前头像接口 `POST /api/auth/profile/avatar` 直接把文件写到 `app.upload.dir`。
2. `application.yml` 里默认的上传目录是绝对路径 `/home/ubuntu/apps/backend-api/uploads`。
3. 在本地开发环境、容器环境或权限不一致的线上环境里，这个目录可能不存在、不可写，导致 `file.transferTo(...)` / 文件写入直接失败。
4. 参考项目 `admin/routes/web-api.js` 的做法是：
   - 使用相对路径 `../public/uploads`
   - 启动时自动创建目录
   - 上传时直接把图片落盘到该目录
   - 返回可访问 URL

结论：
- 当前问题最可能是上传目录不稳定或不可写。
- 需要把“目录解析 + 自动创建 + 回退路径”抽成统一能力，确保头像上传、管理端上传、静态资源读取三处使用同一目录。

## 二、参考实现

参考文件：

- `G:\bishe2 - 副本\admin\routes\web-api.js`

关键参考点：

1. `uploadDir = path.join(__dirname, "../public/uploads")`
2. 不存在则 `fs.mkdirSync(uploadDir, { recursive: true })`
3. `multer.diskStorage` 保存文件时直接用 `uploadDir`
4. 图片上传接口返回 `url: /uploads/{filename}`

这说明参考项目的核心不是“复杂上传协议”，而是：
- 目录固定
- 目录自动创建
- 上传和读取使用同一个目录

## 三、本次修改

### 1) 新增统一上传目录解析组件

文件：

- `G:\zhiximini\backend-api\src\main\java\com\zhixi\backend\config\UploadStorageSupport.java`

新增能力：

1. 优先使用配置项 `app.upload.dir`
2. 如果当前目录不可写，自动回退到：
   - `uploads`
   - `public/uploads`
   - 系统临时目录 `zhixi-backend-uploads`
3. 自动 `createDirectories`
4. 缓存最终选中的可写目录
5. 记录日志，便于定位最终使用的是哪个目录

作用：
- 把“找目录”和“创建目录”从各个 Controller 里抽出来，避免不同接口各自用不同路径。

### 2) 头像上传接口改为使用统一目录解析 + 流式复制

文件：

- `G:\zhiximini\backend-api\src\main\java\com\zhixi\backend\controller\AuthController.java`

修改点：

1. 注入 `UploadStorageSupport`
2. 头像上传时，不再直接依赖 `Paths.get(uploadDir)`
3. 目录通过 `uploadStorageSupport.resolveUploadDir()` 获取
4. 文件落盘改为 `Files.copy(file.getInputStream(), target, REPLACE_EXISTING)`
5. 捕获异常时记录目标路径和原始异常，返回更具体的错误信息

作用：
- 避免 `transferTo` 在目录异常/路径切换时失败
- 当上传失败时，前端能拿到更具体的报错

### 3) 管理端上传接口同步使用统一目录解析

文件：

- `G:\zhiximini\backend-api\src\main\java\com\zhixi\backend\controller\AdminUploadController.java`

修改点：

1. 注入 `UploadStorageSupport`
2. 保存目录改为统一的可写目录解析结果
3. 文件写入改为 `Files.copy(...)`
4. 保留原有图片大小校验

作用：
- 让管理端图片上传和头像上传完全走同一套落盘逻辑

### 4) 公共图片读取接口改为读取统一目录

文件：

- `G:\zhiximini\backend-api\src\main\java\com\zhixi\backend\controller\UploadPublicController.java`

修改点：

1. 注入 `UploadStorageSupport`
2. 读取文件时使用 `uploadStorageSupport.resolveFile(filename)`

作用：
- 保证上传保存的目录和访问图片的目录一致
- 避免“上传成功但读不到”的路径不一致问题

### 5) Spring MVC 静态资源映射改为读取统一目录

文件：

- `G:\zhiximini\backend-api\src\main\java\com\zhixi\backend\config\WebConfig.java`

修改点：

1. 注入 `UploadStorageSupport`
2. `/api/uploads/**` 的资源位置改为 `uploadStorageSupport.resolveResourceLocation()`

作用：
- 让静态资源映射和上传落盘目录保持一致
