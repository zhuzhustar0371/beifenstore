## 一、问题分析（修改前）

根据前端控制台截图与代码排查，问题点如下：

1. 前端 `pages/login/login.js` 在 `onEnterProgram` 中请求 `POST /api/auth/profile`，接口报 500。
2. 后端接口链路:
   - `AuthController#updateProfile` -> `UserAuthService#updateCurrentUserProfile`
   - 该方法会调用 `userWechatAuthMapper.updateProfile(...)`，SQL 依赖 `source_type/unionid/avatar_url/updated_at`。
3. 若线上 `user_wechat_auth` 表是旧结构（缺字段），`updateProfile` 会抛 SQL 异常，落入全局异常处理并返回 500。
4. 前端当前是“后端保存成功才进入程序”，所以 500 会直接卡住流程。

## 三、本次代码修改

### 1) 后端数据库迁移补强

文件：`backend-api/src/main/java/com/zhixi/backend/config/DatabaseMigrationRunner.java`

新增迁移兜底：

- `user_wechat_auth.unionid`
- `user_wechat_auth.avatar_url`

目的：线上表若缺字段，启动时自动补齐，避免 `updateProfile` SQL 直接失败。

关键位置：
- `DatabaseMigrationRunner.java:96`
- `DatabaseMigrationRunner.java:101`

### 2) 后端 Mapper 增加兼容 SQL

文件：`backend-api/src/main/java/com/zhixi/backend/mapper/UserWechatAuthMapper.java`

新增方法：

- `updateNicknameOnly(id, nickname)`
- `insertLegacy(userId, openid, nickname)`

用途：当完整 SQL 因旧表字段缺失失败时，回退到旧表可执行的最小 SQL，避免 500。

关键位置：
- `UserWechatAuthMapper.java:36`
- `UserWechatAuthMapper.java:40`

### 3) 后端保存资料增加容错回退

文件：`backend-api/src/main/java/com/zhixi/backend/service/UserAuthService.java`

在 `updateCurrentUserProfile(...)` 中新增两层容错：

- `updateProfile` 失败 -> 回退 `updateNicknameOnly`
- `insert` 失败 -> 回退 `insertLegacy`

并增加 warning 日志，便于线上排查真实根因。

关键位置：
- `UserAuthService.java:502`
- `UserAuthService.java:523`

### 4) 前端请求失败本地兜底

文件：`wechat-app/pages/login/login.js`

在 `onEnterProgram` 的 catch 中新增逻辑：

- 若 token 仍在（说明不是未登录），则把昵称和头像先写本地缓存 `userInfo`
- Toast 提示“资料同步失败，已先进入程序”
- 继续 `switchTab('/pages/user/user')`

效果：后端瞬时 500 不再阻断用户进入。

关键位置：
- `login.js:18`
- `login.js:165`
- `login.js:203`

### 5) “我的”页昵称合并策略修复

文件：`wechat-app/pages/user/user.js`

新增昵称合并规则：

- 若后端返回默认昵称（如“微信用户xxxx”），且本地已有用户自定义昵称，则保留本地昵称
- 避免“刚填写昵称后又被默认昵称覆盖”

关键位置：
- `user.js:72`
- `user.js:82`
- `user.js:98`

并同时修复该文件历史编码污染（统一为可读中文文案，移除 BOM）。
