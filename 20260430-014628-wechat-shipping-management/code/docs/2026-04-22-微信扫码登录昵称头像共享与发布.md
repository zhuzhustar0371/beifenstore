# 2026-04-22 微信扫码登录昵称头像共享与发布日志

## 一、需求背景

本次需求包含两部分：

1. Web 端微信扫码登录与小程序微信授权登录，需要共用同一个昵称和头像。
2. 确认并落地 Web 端微信扫码登录能力，避免“扫码后只有会话授权，没有对应用户资料”的情况。

在分析后确认：

- Web 扫码登录流程本身已经存在，核心入口在 `zhixi-website/frontend/src/components/LoginModal.vue`。
- 现有问题主要是后端未把 Web / 小程序两条微信身份链路的昵称与头像做统一同步，前端用户中心和顶部登录态也没有把头像完整展示出来。

## 三、分析结论

本次问题的本质不是“Web 扫码登录没有创建用户”，而是：

- Web / 小程序各自拿到的微信昵称和头像没有统一同步到同一份用户资料；
- 前端展示页和顶部登录态只展示了昵称，没有把头像渲染出来；
- Web 扫码登录虽然已有 token 与用户创建链路，但用户视觉上看起来像“只扫码未登录成功”。

因此本次修复采取的方案是：

- 后端统一 Web / 小程序微信资料同步逻辑；
- 登录后把昵称和头像写回共享资料；
- 前端按头像优先、文本兜底的方式展示；
- 保持现有 Web 扫码登录流程不变，只补齐资料统一与展示。

## 四、代码修改

### 4.1 后端：统一微信资料读取与同步

文件：

- `G:\zhiximini\backend-api\src\main\java\com\zhixi\backend\service\UserAuthService.java`

主要改动：

1. 新增统一资料解析逻辑 `resolveWechatProfile(...)`，优先从当前用户的 Web / 小程序微信授权记录中解析共享昵称和头像。
2. `getUserAvatarUrl(...)` 和 `getDisplayNickname(...)` 改为读取共享资料，不再只偏向单一来源。
3. 在以下场景加入 `syncWechatProfile(...)`，让 Web 与小程序两侧的资料保持一致：
   - 小程序登录后
   - Web 扫码 OAuth 回调后
   - 用户主动更新资料后
4. 同步逻辑会同时更新：
   - `user` 主表中的昵称
   - `user_wechat_auth` 的 `WEB` 记录
   - `user_wechat_auth` 的 `MINIAPP` 记录

这样可以保证：

- Web 扫码登录后创建或绑定的用户，能继承/统一头像昵称；
- 小程序授权登录后，Web 端也能看到同一份头像昵称；
- 两端最终展示同一用户资料。

### 4.2 前端：用户中心展示头像

- `G:\zhiximini\zhixi-website\frontend\src\views\UserCenterPage.vue`

3. 保留昵称和手机号脱敏显示，不影响现有页面布局。
