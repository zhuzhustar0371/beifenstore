- 目标：将线上商品测试金额调整为 `0.10` 元，并让返现金额按实际订单金额等比例缩小，便于微信支付真实下单测试。

## 变更前状态

上一轮测试中，线上商品价格已临时调整为：

- `0.01` 元

后端返现逻辑仍是旧逻辑：

- 个人第 2 单固定返 `9.90`
- 个人第 3 单固定返 `19.80`
- 个人第 4 单固定返 `69.30`
- 邀请第 1 批固定返 `99.00`
- 邀请后续批次固定返 `20.00`

该逻辑在低金额支付测试中会产生不合理返现数据。

## 本次代码变更

### 1. 后端个人返现改为按实际订单金额计算

修改文件：

- `backend-api/src/main/java/com/zhixi/backend/service/CashbackService.java`
- `backend-api/src/main/java/com/zhixi/backend/service/OrderService.java`

新逻辑：

- 第 1 单：不返
- 第 2 单：订单金额 × 10%
- 第 3 单：订单金额 × 20%
- 第 4 单：订单金额 × 70%
- 第 5 单及以后：不返

当测试订单金额为 `0.10` 元、数量为 `1` 时，预期为：

- 第 2 单：`0.01` 元
- 第 3 单：`0.02` 元
- 第 4 单：`0.07` 元

### 2. 邀请返现按测试价等比例缩小

新逻辑：

- 第 1 批满 3 人：按当前触发订单金额结算
- 第 2 批及以后：按原 `20 / 99` 的比例乘以当前触发订单金额，四舍五入到分

当触发订单金额为 `0.10` 元时，预期为：

- 第 1 批：`0.10` 元
- 第 2 批：约 `0.02` 元
- 第 3 批及以后：约 `0.02` 元

### 3. 初始化数据同步为测试价

修改文件：

- `backend-api/src/main/resources/data.sql`

默认商品价改为：

- `0.10`

默认商品描述改为：

- `单价0.10元/件，温和洁净，适合家庭日常使用。`

### 4. 小程序规则页同步展示测试金额

修改文件：

- `wechat-app/pages/rules/rules.wxml`

展示更新：

- 商品单价：`0.10` 元/件
- 第 2 单返现：`0.01` 元
- 第 3 单返现：`0.02` 元
- 第 4 单返现：`0.07` 元
- 邀请第 1 批：`0.10` 元
- 邀请后续批次：`0.02` 元

## 测试计划

### 支付链路测试

1. 微信开发者工具或真机重新进入小程序
2. 进入商品详情页
3. 确认商品价格为 `0.10`
4. 数量保持 `1`
5. 创建新订单
6. 确认订单金额为 `0.10`
7. 点击支付
8. 预期弹起微信支付，金额为 `0.10`

### 支付回调测试

支付完成后检查：

- 微信支付回调是否到达 `/api/pay/wechat/notify`
- 订单状态是否从 `PENDING` 变为 `PAID`
- `pay_type` 是否为 `WECHAT`
- `transaction_id` 是否写入
- `pay_time` 是否写入

### 返现测试

使用同一用户连续完成真实支付订单：

- 第 1 单：不返
- 第 2 单：返 `0.01`
- 第 3 单：返 `0.02`
- 第 4 单：返 `0.07`
- 第 5 单及以后：不返

邀请返现测试需要准备 3 个被邀请用户完成首单支付：

- 第 1 批满 3 人：返 `0.10`
- 第 2 批满 3 人：返 `0.02`
- 后续批次：每批返 `0.02`

## 注意事项

- 已有历史订单金额不会被修改
- 已有历史返现记录不会被重算
- 新逻辑只影响后续支付成功后生成的新返现记录
- 小程序规则页文件已更新，但需要重新预览/上传小程序代码才能让用户端看到最新规则页
- 线上商品接口已经是 `0.10`

## 回滚方式

如需恢复正式价和正式返现规则，需要：

1. 将商品价格恢复为 `99.00`
2. 将商品描述恢复为 `单价99元/件，温和洁净，适合家庭日常使用。`
3. 如果希望回到固定返现逻辑，则回滚 `CashbackService.java` 和 `OrderService.java`
4. 重新构建并部署后端
5. 恢复小程序规则页展示金额

如果保留“按实际订单金额比例返现”的新逻辑，则恢复商品价到 `99.00` 后，个人返现会自然回到：

- 第 2 单：`9.90`
- 第 3 单：`19.80`
- 第 4 单：`69.30`
