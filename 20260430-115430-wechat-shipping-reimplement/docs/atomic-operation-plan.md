## 问题原子化拆解

1. 管理端点击“发货”后，本地订单如果只改成本地 SHIPPED，而没有调用微信官方发货信息管理接口，则微信小程序官方购买记录不会进入“待收货”，资金结算也可能仍被冻结。
2. 微信官方发货接口查找订单依赖支付单唯一键：优先微信支付单号 transaction_id，或者商户号 mchid + 商户订单号 out_trade_no；本地订单必须把这些字段从支付回调保存并透出。
3. 当前工作区已有部分发货管理实现与新增改动：公共 MiniappAccessTokenService、发货接口 token 失效重试、小程序确认收货回调用 pending order id 兜底匹配。
4. 风险点：access_token 多处重复获取会互相顶掉；确认收货组件返回后若列表过滤了订单，可能找不到本地订单；微信接口若返回支付单不存在，需要把尝试过的 key 暴露给管理端排查。
5. 发布风险：后端健康检查需要足够等待时间，避免服务实际启动较慢时误判失败。

## 解决方案逻辑链

1. 备份当前稳定/当前待改源码到本地和远端，确保可回滚。
2. 后端统一小程序 access_token 获取，使用 stable_token，并在 40001/42001 等 token 失效场景强制刷新后重试一次。
3. 后端发货前先用 transaction_id 与 mchid+out_trade_no 依次查询微信订单状态，匹配成功后使用对应 order_key 上传发货信息。
4. 若微信订单已发货，按幂等成功处理，允许本地订单继续进入 SHIPPED，避免重复发货阻塞。
5. 后端设置微信发货消息跳转路径到小程序订单列表待收货 tab。
6. 小程序端确认收货按钮不只依赖 transaction_id，也支持 mchid+orderNo；打开官方组件前记录本地 pending order id，回调后再调用后端 sync 接口确认微信状态。
7. 管理端显示 payType、transaction_id、out_trade_no，发货失败时给出可排查信息。
8. 编译/静态检查，必要时部署；如部署失败按备份 jar / 备份源码回退。
