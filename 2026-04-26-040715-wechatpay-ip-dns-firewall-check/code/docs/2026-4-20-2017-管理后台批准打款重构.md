## 问题分析

核查现有代码后，发现“批准打款”链路有四个实际问题：

1. 前端虽有按钮实现，但线上旧版页面可能未部署，截图中仍没有操作列。
2. 后端返现打款接口复用了 `cashbacks:read` 权限，读写没有拆开。
3. 微信公钥模式只支持文件路径，不支持用户直接提供 PEM 内容，部署时不够灵活。
4. 打款时会生成 `out_batch_no` 和 `out_detail_no`，但原实现没有写回数据库，后续查询批次、对账和回调关联会缺少关键字段。

## 本次修改

### 1. 后端支持直接使用微信支付 PEM 公钥

修改文件：

- `backend-api/src/main/java/com/zhixi/backend/service/WechatPayService.java`
- `backend-api/src/main/resources/application.yml`
- `backend-api/.env.example`

修改内容：

- 新增配置项 `app.wechat.pay.public-key-pem`
- 对应环境变量：`WECHAT_PAY_PUBLIC_KEY_PEM`
- 保留原有 `WECHAT_PAY_PUBLIC_KEY_PATH`
- 当未提供公钥文件路径、但提供了 PEM 内容时：
  - 后端将 PEM 内容写入临时文件
  - 再使用 `RSAPublicKeyConfig` 初始化微信支付 SDK
- 这样可以兼容两种部署方式：
  - 服务器挂载公钥文件
  - 环境变量直接注入 PEM

说明：

- 本次没有把用户提供的完整公钥内容硬编码进仓库
- 代码只增加了“支持直接接收 PEM 内容”的能力
- 线上部署时可将用户提供的公钥写入 `WECHAT_PAY_PUBLIC_KEY_PEM`

### 2. 管理后台返现权限拆分为读写

修改文件：

- `backend-api/src/main/java/com/zhixi/backend/service/AdminPermissionService.java`
- `backend-api/src/main/resources/data.sql`

修改内容：

- 新增权限：`cashbacks:write`
- `/api/admin/cashbacks`：
  - `GET` 仍要求 `cashbacks:read`
  - `POST` 等写操作改为要求 `cashbacks:write`
- `SUPER_ADMIN` 默认授予 `cashbacks:write`
- `OPERATOR` 仍只保留 `cashbacks:read`

结果：

- 返现查看和返现打款不再混用同一权限
- “批准打款”具备独立写权限控制

### 3. 返现打款批次号和明细号落库

修改文件：

- `backend-api/src/main/java/com/zhixi/backend/mapper/CashbackRecordMapper.java`
- `backend-api/src/main/java/com/zhixi/backend/service/AdminManageService.java`

修改内容：

- 原实现只更新：
  - `status`
  - `transfer_id`
  - `transfer_time`
- 现改为同时落库：
  - `out_batch_no`
  - `out_detail_no`
  - `transfer_id`
  - `transfer_time`
  - `status`

结果：

- 微信商家转账发起后，返现记录中会保留微信批次关联号
- 后续查询单笔打款、批次排障、对账时有完整主键链路

### 4. 重做管理后台返现管理页

修改文件：

- `zhixi-website/admin-frontend/src/App.vue`

修改内容：
