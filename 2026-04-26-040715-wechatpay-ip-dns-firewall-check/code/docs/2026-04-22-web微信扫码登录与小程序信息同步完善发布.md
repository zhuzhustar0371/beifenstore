# 2026-04-22 Web 微信扫码登录与小程序信息同步完善发布日志

## 1. 需求背景

本次工作的目标有两项：

1. 继续完善 Web 端微信扫码登录，确保流程稳定可用。
2. 确保 Web 端与小程序端的微信用户信息保持同步，避免昵称、头像和 openid 使用混乱。

在分析现有代码后，确认并不是“完全没有实现”，而是存在以下需要收口的问题：

- Web 微信扫码登录已经有标准 OAuth 回调链路，但二维码展示依赖第三方 `quickchart.io`。
- 后端已经有昵称和头像共享逻辑，但旧的 `getDisplayNickname(...)` 还保留了历史分支。
- 旧接口 `bindWechatMobile(...)` 仍会把 Web `openid` 写入 `miniappOpenid`，存在污染返现与支付标识的风险。

## 3. 代码分析结论

### 3.1 Web 微信扫码登录现状

后端已具备以下能力：

- 生成 `qrconnect` 授权 URL
- 持久化扫码场景 `scene`
- 微信回调接收 `code` 和 `state`
- 通过 `code` 换取 `access_token/openid/unionid`
- 通过 `sns/userinfo` 获取昵称与头像
- 创建或绑定本地用户并生成系统 token

前端已具备以下能力：

- 登录弹窗中发起“微信扫码登录”
- 获取扫码场景和授权 URL
- 轮询扫码状态
- 登录成功后写入 token 并拉取当前用户资料

### 3.2 需要修复的问题

确认本次需要修复的点为：

1. 去掉二维码生成对第三方 `quickchart.io` 的依赖。
2. 将旧的昵称读取入口统一转向共享昵称逻辑。
3. 修复 `bindWechatMobile(...)` 误写 `miniappOpenid` 的问题。

## 4. 本地代码修改

### 4.1 后端修改

修改文件：

- [UserAuthService.java](G:/zhiximini/backend-api/src/main/java/com/zhixi/backend/service/UserAuthService.java)

具体修改：

1. `createWechatQrSession()` 不再拼接第三方二维码图片地址：
   - 原先返回 `quickchart.io` 二维码图片 URL
   - 现在仅返回授权 URL，`qrImageUrl` 置空，交由前端本地生成二维码

2. `getDisplayNickname(...)` 统一改为调用共享逻辑：
   - 直接转向 `getSharedDisplayNickname(...)`
   - 避免后续接口误用旧逻辑，导致昵称显示再次偏向单端来源

3. `bindWechatMobile(...)` 修复 Web openid 污染问题：
   - 删除 `userService.updateMiniappOpenid(user.getId(), session.getOpenid())`
   - 改为调用 `syncWechatProfile(...)`
   - 确保 Web 绑定只同步昵称，不再把 Web openid 错误写入小程序标识字段

### 4.2 前端修改

修改文件：

- [LoginModal.vue](G:/zhiximini/zhixi-website/frontend/src/components/LoginModal.vue)

具体修改：

1. 引入已有依赖 `qrcode`
2. 将 `buildWechatQrImageUrl(...)` 从同步字符串拼接改为本地二维码生成：
   - 使用 `QRCode.toDataURL(...)`
   - 不再依赖第三方二维码服务
3. `startWechatLogin()` 中改为本地生成二维码图片

本次未新增 npm 依赖，直接复用项目原有的：

- `qrcode`

该依赖此前已在支付二维码中使用。
