## 1. 问题现象

- 用户反馈：Web 端微信扫码登录后，浏览器停留在 `https://api.mashishi.com/api/auth/wechat/callback?code=...&state=...` 页面，无法自动跳回到已登录的网站页面。
- 直接影响：callback 页只显示成功提示，不会自动回跳，体验上像“登录卡住”。

## 2. 修改前分析结论

- 后端 `GET /api/auth/wechat/callback` 只返回静态 HTML，没有 302 重定向，也没有自动回跳脚本。
- Web 扫码登录的主浏览器依赖前端轮询 `scene` 状态完成 token 写入，所以 callback 页本身不负责主站 token 同步。
- 当前实现缺少“登录成功后自动回到前台网站”的收口动作，因此 callback 页面会停留在 `api` 域名。
- 仓库中没有现成的 `WECHAT_WEB_SUCCESS_REDIRECT_URI` 配置项，回跳地址无法通过环境变量独立控制。

## 12. 二次问题确认

- 用户反馈：页面虽然已经从 `api.mashishi.com` 返回首页，但首页仍然显示未登录。
- 复盘结论：
  - 原先的修复只解决了“页面回跳”，没有解决“登录态传递”。
  - 前台登录态实际保存在 `mashishi.com` 域下的 `localStorage`，而 callback 页运行在 `api.mashishi.com` 域，二者不能直接共享本地存储。
  - 因此，仅做 `window.location.replace` 回首页并不能让前台自动识别为已登录。

## 13. 第二轮修改

### 13.1 修改文件

- `G:\zhiximini\backend-api\src\main\java\com\zhixi\backend\controller\AuthController.java`
- `G:\zhiximini\zhixi-website\frontend\src\main.js`

### 13.2 修改内容

1. 后端 callback 成功页不再只回跳首页，而是将登录 `token` 编入前台回跳地址的 `hash` 段。
2. 前端入口在应用挂载前读取 `window.location.hash`，识别 `wechatLoginToken` 后立即写入 `localStorage`。
3. 前端随后清理地址栏里的临时 `hash`，避免 token 长时间暴露在 URL 中。
4. 这样回到前台首页后，`App.vue` 的启动检查可以直接拿到 token 并拉取用户信息，页面会显示为已登录。
