# Product Strategy - MVP 边界与验证标准

> **战略目标**：在区/县范围内验证 `发布 → 浏览 → 联系 → 回复` 闭环，防止功能膨胀。

---

## 1️⃣ P0 / P1 / P2 功能确认

### 📍 P0 - 核心闭环（必须实现）

| 功能域 | P0 范围 | 理由 |
|--------|--------|------|
| **用户认证** | 微信登录 | 完成"我是谁"的身份确认 |
| **商品发布** | 标题/描述/价格/图片/区县 | 用户生产内容的最小集 |
| **商品浏览** | 列表（分区县、搜索关键词）+ 详情页 | 用户发现内容的最小集 |
| **联系交流** | 发起会话、发送/接收文字消息 | 完成买卖双方沟通的最小集 |
| **内容审核** | 管理员通过/拒绝商品、下架商品 | 平台质量把控的最小集 |
| **用户管理** | 禁用违规用户 | 平台安全的最小集 |
| **反馈收集** | 用户提交反馈 + 管理员查看 | 种子用户意见的最小集 |

**P0 页面总数**：用户端 10 页 + 后台 4 页 = **共 14 个页面**

---

### 📍 P1 - 体验优化（第二阶段可考虑）

| 功能 | 理由 | 优先级 |
|------|------|--------|
| 发布后编辑/删除 | 用户误操作纠正 | 高 |
| 图片多选/轮播 | 提案呈现效果 | 中 |
| 区县记忆/默认 | 减少重复选择 | 中 |
| 消息未读 badge + 通知 | 用户触达 | 高 |
| 简化个人资料修改 | 减少账户管理 | 低 |
| 商品置顶/排序 | 可选，非核心 | 低 |
| 收藏（功能） | 非闭环必需 | 低 |

---

### 📍 P2 - 明确禁区（当前严禁，留待未来）

| 功能 | 原因 | 后期考虑 |
|------|------|---------|
| **支付/订单** | 引入资金风险、技术复杂度激增 | 数据验证充分后的 1.0 版本 |
| **担保交易** | 需要资金托管，法规风险大 | 用户规模 > 1000 且融资后 |
| **收货地址管理** | 增加隐私风险和交付复杂度 | 支付系统上线 |
| **客服工作台** | 系统复杂度高，前期手工足够 | 投诉量 > 50 件/周 |
| **推荐/信用体系** | 需大数据支撑，冷启动效果差 | 数据积累 > 500 条商品 |
| **多城市管理** | 扩大运营成本 | 单市场验证充分后 |
| **商业化功能** | 当前应该"让用户爽" 而非"变现" | MVP 结束后考虑 |

## 2️⃣ 第一阶段"明确不做清单"

## 3️⃣ MVP 关键验证指标（5 项）

这 5 项指标用于衡量"这个 MVP 是否足以验证假设"。

### 📊 指标 1：内容供给量

**指标**：首批种子期内的商品发布数  
**目标**：≥ **50 条** 商品记录  
**为什么**：足够组成一个小市场，能让买家有选择、有浏览动力  
**检查点**：发布后商品状态分布（待审核 vs 已上架 vs 已下架）

---

### 💬 指标 2：闭环转化率

**指标**：发布商品转化为会话的比例  
**目标**：≥ **30 %**（50 条发布 × 30 % ≈ 15 条会话）  
**为什么**："联系率"是衡量内容吸引力和平台价值的核心指标  
**检查点**：`会话数 / 已上架商品数`，如果 < 20 % 则说明详情页缺乏吸引力

---

### ⚡ 指标 3：核心功能可用性

**指标**：用户在无崩溃的前提下完成闭环的比例  
**目标**：≥ **80 %** 的登录用户能完整走通"发布 → 浏览 → 联系"  
**为什么**：如果只有 50 % 的人能成功，说明产品质量不过关  
**检查点**：
- 登录成功率 ≥ 95 %
- 发布完成率 ≥ 90 %
- 消息发送成功率 ≥ 95 %

---

### ✅ 指标 4：审核响应速度

**指标**：管理员从接到商品到审核的平均时间  
**目标**：< **24 小时**（更好的是 < 4 小时）  
**为什么**：审核速度直接影响发布者体验，影响闭环完成  
**检查点**：
- 是否有卡在"待审核"的僵尸商品
- 管理员是否理解审核标准
- 是否给了用户拒绝原因

---

### 💡 指标 5：用户反馈质量

**指标**：用户反馈的数量与正向占比  
**目标**：≥ **10 条** 反馈，其中 ≥ **60 %** 包含建议或积极评价  
**为什么**：反馈数量说明用户投入，反馈质量说明用户思考  
**检查点**：
- 最高频的问题是什么（可能需要紧急修复）
- 是否有"我想继续用"这类反馈
- 反馈来自多少个不同用户（≥ 5 个为好）

---

## 4️⃣ 首批种子用户验证判断标准

### ✨ 通过标准（同时满足以下 4 条）

#### 通过条件 1：完整闭环验证 ✅

- 至少 **3** 个真实用户完整走通"发布 → 浏览 → 联系 → 回复"全过程
- 其中至少 **1** 个用户是"卖家身份发布"并"买家身份联系"的双身份体验
- 验证方法：进后台查询 `conversations` 集合，看双向消息记录

---

#### 通过条件 2：无阻塞性问题 ✅

- 整个测试期，**没有导致无法使用的 bug**（崩溃、黑屏、接口超时等）
- 修复后能重现通过闭环
- 性能指标：
  - 登录成功率 ≥ 95 %
  - 发布成功率 ≥ 90 %
  - 消息发送成功率 ≥ 95 %
- 允许的非阻塞问题：UI 排版偏差、图片加载稍慢等

---

#### 通过条件 3：用户积极反馈 ✅

- 至少有 **1** 条来自种子用户的**积极或建设性反馈**（如"这个功能我喜欢"、"建议能加上..."）
- **没有**"这个软件没法用"这类绝对否定的反馈
- 收集方法：通过"反馈页面"、问卷、或面访

---

#### 通过条件 4：后台控制力验证 ✅

- 管理员可以**审核通过**一条商品，该商品在 **5 分钟内**在用户端可见
- 管理员可以**下架**一条商品，该操作在 **5 分钟内**在用户端生效（用户无法联系）
- 管理员可以**禁用**一个用户，该用户**无法继续发新商品或发新消息**
- 目的：验证平台有基础的内容治理能力

---

### ❌ 失败/阻塞标准

| 失败信号 | 处理方案 |
|----------|---------|
| 无法完成 1 个完整闭环 | 找阻塞点，修复后重新测 |
| ≥ 2 个用户反馈"无法登录" | 紧急修复登录逻辑 |
| ≥ 2 个用户反馈"发布商品失败" | 紧急修复发布接口 |
| 管理员审核/下架功能不工作 | 重做后台页面/接口 |
| 没有任何用户反馈或全是"不好用" | UX 或设计有根本问题 |
| 会话消息收不到 / 延迟 > 10 秒 | 排查云函数或数据库权限 |

## 📋 团队协作模式

### 需求变更流程

1. **新需求提出** → 首先对标 P0 边界
2. **是否阻塞闭环**❓
   - 是 → P0，立即修复
   - 否 → 记录到 P1，排期到第二阶段
3. **严禁在冲刺期添加 P2 功能**

### Bug 优先级划分

| 优先级 | 定义 | 处理 |
|--------|------|------|
| P0-Bug | 阻塞闭环（无法登录/发布/发消息/审核） | 立即停工修复 |
| P1-Bug | 影响体验但不打断闭环（UI 错位、加载慢） | 排期修复 |
| P2-Bug | 非核心功能的问题 | 记录待处理 |

---

## 🎯 成功验证清单

当达到以下状态，说明"第一阶段产品验证成功"：

- ✅ ≥ 50 条商品发布
- ✅ ≥ 15 条会话（≥ 30 % 转化）
- ✅ 80 % 用户无阻塞完成闭环
- ✅ 商品审核 < 24 小时返回
- ✅ ≥ 10 条有质量反馈，60 % 正向
- ✅ 3 个真实用户完整体验，1 个双身份体验
- ✅ 无阻塞性 bug，后台控制力可验证
- ✅ ≥ 1 条积极反馈

---

## 🚀 后续行动

验证成功后，可以：

1. **总结报告** — 向投资人/利益相关者展示数据
2. **迭代 P1 功能** — 编辑删除、图片轮播、消息通知等
3. **扩展市场** — 复制到其他城市
4. **融资探索** — 基于数据讨论商业化
