## 1. 审查结论

当前“冻结契约”还**没有完全对齐**，主要存在 4 类分歧：

1. `openid` 与 `open_id` 两套命名并存。
2. `listings.image_urls` 与 `listing_images` 两套图片来源并存。
3. `messages` 被部分文档写成子集合，部分文档写成独立集合。
4. `02-data-model.md` / 管理端代码仍使用更重的数据模型，超出 Prompt 3 的最小联调范围。

本轮建议冻结为：

- **前端联调最小契约**：以 `LAUNCH-ALL-AGENTS.md` 的 Prompt 3 为主。
- **后台兼容扩展**：允许后台保留额外字段，但前端和 Data Infra 不依赖它们。
- **CloudBase 主键统一**：数据库实际主键为 `_id`，对前端 DTO 映射为 `id`。
- **业务身份统一命名**：统一使用 `openid`，不再新增 `open_id` / `seller_id` / `buyer_id` / `user_id` 作为联调主字段。

## 2. 7 个集合的最终字段定义

| 集合 | 联调必备字段 | 后台兼容/扩展字段 | 冻结结论 |
|------|--------------|-------------------|----------|
| `users` | `_id(id)` / `openid` / `nickname` / `avatar_url` / `role` / `status` / `created_at` | `updated_at` | 以 `openid` 为唯一业务身份；`role='admin'` + `status='active'` 为管理员判定条件 |
| `districts` | `code` / `name` / `city_code` / `city_name` | `province_code` / `province_name` / `is_active` / `sort_order` | 前端只依赖前 4 个字段 |
| `listings` | `_id(id)` / `openid` / `title` / `description` / `price` / `district_code` / `status` / `image_urls` / `created_at` | `updated_at` / `review_status` / `reject_reason` / `view_count` / `contact_count` | 用户端主数据源；卖家身份字段冻结为 `openid` |
| `listing_images` | `_id(id)` / `listing_id` / `image_url` / `sort_order` | 无 | 保留为兼容集合；**不是**用户端主读源 |
| `conversations` | `_id(id)` / `listing_id` / `buyer_openid` / `seller_openid` / `last_message` / `updated_at` / `created_at` | `buyer_unread_count` / `seller_unread_count` / `status` | 物理存储建议按买卖双方分别记录未读数；前端可映射出单个 `unread_count` |

| `feedback` | `_id(id)` / `openid` / `category` / `content` / `contact_info` / `created_at` | `status` | 前端提交只需最小字段；后台若继续筛选反馈状态，可保留 `status='new'` |

### 补充说明

- `listing_images` 保留，但本轮联调中**主存储方案冻结为 `listings.image_urls`**。
- `conversations.unread_count` 单字段定义不够精确，实际存储建议拆成 `buyer_unread_count` 和 `seller_unread_count`；前端展示时再映射成当前用户的 `unread_count`。
- `feedback.status` 不属于前端最小写入契约，但当前管理端依赖它筛选，后台可默认补写 `new`。

## 3. 4 个页面的最终数据约束

| 页面 | 读写对象 | 最终约束 | 冻结结论 |
|------|----------|----------|----------|
| 首页 | `listings` | `WHERE status='approved'`；可选 `district_code` 过滤；标题支持 `keyword` 模糊匹配 | 首页只读审核通过商品 |
| 详情页 | `listings` + `users`（可选） + `listing_images`（兼容回退） | 先查 `listings`；图片优先读 `image_urls`；仅在兼容期 `image_urls` 为空时回退查 `listing_images` | 用户端详情主数据源是 `listings` |
| 发布页 | `listings` | 新建记录必须写入 `status='pending_review'`；图片写入 `image_urls` 数组 | 禁止前端直接写 `approved` / `removed` / `rejected` |
| 消息页 | `conversations` + `messages` | 会话查询条件：`buyer_openid=currentOpenid OR seller_openid=currentOpenid`；消息查询条件：`conversation_id=...`，按 `created_at ASC` | `messages` 独立集合，不走子集合 |

## 4. 已冻结的关键设计

### 4.1 listings.image_urls 的最终方案

- **冻结结论**：`listings.image_urls` 是商品图片的**主存储字段**。
- **前端读取**：默认只读 `image_urls`。
- **兼容策略**：`listing_images` 仅用于兼容当前后台或历史数据；若继续保留，必须与 `image_urls` 保持一致。

### 4.2 conversations / messages 的关系设计

- **冻结结论**：`messages` 是**独立集合**，通过 `conversation_id` 外键关联 `conversations`。
- **不采用**：`conversations/{id}/messages` 子集合方案。
- **发送消息时必须同步更新**：
  - `conversations.last_message`
  - `conversations.updated_at`
  - 对应接收方未读计数

### 4.3 admin 鉴别机制

- **冻结结论**：管理员身份由 `users.role='admin'` 且 `users.status='active'` 判定。
- **运行时鉴权位置**：仅管理端 Express 做管理员鉴权。
- **用户端页面**：首页 / 详情 / 发布 / 消息 不做管理员鉴权。
- **`.env` 中的 `ADMIN_OPEN_ID`**：仅作为初始化便利项，不是运行时权限来源。

## 5. 前后端必须共同遵守的 3 个关键约束

1. 发布商品默认 `status='pending_review'`。
2. 首页只读 `status='approved'` 的商品。
3. 管理端审核通过后，将 `listings.status` 更新为 `approved`，前端 5 分钟内可见。

## 6. 当前不一致点与修改建议

### P0 必改

1. `execution-plan-round1.md` 仍把 `messages` 写成子集合，应统一改成独立集合 + `conversation_id` 关联。  
   影响位置：第二章核心约束、第一步说明、验收闭环描述。

2. `execution-plan-round1.md` 与 `LAUNCH-ALL-AGENTS.md` 对图片来源描述不一致。  
   应统一为：`listings.image_urls` 是主方案，`listing_images` 仅兼容保留。

3. `02-data-model.md`、`init-db.js`、管理端路由、用户查询脚本仍大量使用 `open_id` / `seller_id` / `buyer_id` / `user_id`。  
   应统一到本契约的 `openid` / `buyer_openid` / `seller_openid`。

4. `.env` 当前并没有 `ADMIN_OPEN_ID`，但 `LAUNCH-ALL-AGENTS.md` Prompt 2 默认它已存在。  
   应补充该变量，或同步修改 Prompt 2 的初始化说明。

### P1 建议尽快处理

1. `conversations.unread_count` 建议改为物理存储 `buyer_unread_count` + `seller_unread_count`，避免单字段语义不清。
2. `feedback.status` 虽未纳入前端最小契约，但后台管理已依赖它，建议在后台创建时统一默认值 `new`。
3. `admin_actions` 不在 Prompt 3 的 7 个核心集合内，但当前后台审核/禁用/反馈状态更新都会写它；初始化脚本应继续保留该集合，直到后台逻辑改造完成。

## 8. 最终结论

本轮联调的**唯一主契约**应按本文执行：

- 前端和 Data Infra 以 **`openid` + `listings.image_urls` + 独立 `messages` 集合** 为主。
- 后台可以保留扩展字段，但不得再反向要求前端依赖 `seller_id` / `user_id` / `open_id` 这套旧命名。
- 若不先完成这次字段收敛，Frontend Dev、Data Infra、Backend Admin 三方会继续按不同数据模型开发，后续必然返工。
