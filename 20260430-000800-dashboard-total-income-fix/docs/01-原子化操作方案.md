# 20260430-000800-dashboard-total-income-fix 原子化操作方案

## 任务目标

- 分析并修复后台看板“总收入”显示不真实的问题。
- 优先排查历史订单字段迁移导致的统计失真。
- 在明确统计口径后修复后端统计逻辑，并做本地验证。

## 当前已知现象

- 管理后台看板显示“总收入 = 0”。
- 同页显示“总返现 = 15”，说明系统中已有返现记录，订单支付链路至少部分执行过。
- 前端仅展示 `/api/admin/dashboard` 返回的 `totalIncome` 字段。

## 已确认代码事实

- 看板接口：`backend-api/src/main/java/com/zhixi/backend/controller/AdminController.java`
- 总收入来源：`orderService.totalPaidAmount()`
- 当前统计 SQL：`SELECT COALESCE(SUM(total_amount),0) FROM orders WHERE order_status IN ('PAID','SHIPPED','COMPLETED')`
- 历史迁移存在风险：`orders.order_status` 新增为 `NOT NULL DEFAULT 'PENDING'`，后续旧字段同步逻辑仅在 `order_status IS NULL` 时执行，可能导致老数据无法从旧 `status` 正确回填。

## 统计口径候选

- 口径 A：交易总流水
  - 已支付订单总额，不扣退款，不扣返现。
- 口径 B：真实净收入
  - 已支付总额减已成功退款金额。
- 口径 C：经营净收入
  - 真实净收入再减返现支出。

## 当前默认修复方向

- 先修复“历史订单状态未正确映射到 `order_status`”这一确定问题。
- 再把看板“总收入”至少修到与实际订单支付数据一致。
- 如果现有表结构缺少“实际退款金额”字段，则先实现可验证的现有最佳口径，并在日志中明确限制。
