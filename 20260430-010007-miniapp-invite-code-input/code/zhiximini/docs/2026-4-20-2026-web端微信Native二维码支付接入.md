# Web 端微信 Native 二维码支付接入

## 修改时间

- 2026-04-20 20:26

## 需求背景

用户要求使用微信支付 skill，并在 web 端仿照小程序端调用支付接口的方式调用微信支付接口。

现有小程序端已经具备真实微信支付链路：

- 小程序调用 `POST /api/orders/{orderId}/pay/wechat-miniapp`
- 后端创建微信 JSAPI/小程序预支付订单
- 小程序收到支付参数后调用 `wx.requestPayment`
- 微信支付回调到 `/api/pay/wechat/notify`
- 后端按回调更新订单支付状态

web 端原逻辑仍调用旧接口 `POST /api/orders/{orderId}/pay`，该接口会直接将订单标记为已支付，属于 mock 支付路径，不符合真实微信支付流程。

## 本次目标

- 保留小程序端支付链路不变。
- 为 web 端新增真实微信 Native 支付下单入口。
- web 端创建订单后调用真实微信支付接口生成二维码。
- 用户用微信扫码支付后，web 端轮询订单状态并在支付成功后刷新页面状态。
- 避免 web 端继续使用旧 mock 支付接口。

## 后端修改详情

### 1. 新增 Native Pay SDK 服务

在 `WechatPayService` 中新增：

- `NativePayService nativePayService`
- 初始化时通过现有微信支付 `Config` 构建 `NativePayService`

这样 Native 二维码支付与小程序 JSAPI 支付共用当前商户号、appid、私钥、公钥/APIv3Key、回调地址等配置。

### 2. 新增 `createNativeOrder`

新增方法：

- `createNativeOrder(Order order, String description)`

核心逻辑：

- 校验订单号不能为空。
- 复用订单号作为微信支付 `outTradeNo`。
- 使用订单金额换算为分。
- 设置币种 `CNY`。
- 设置微信支付回调地址 `notifyUrl`。
- 调用 `nativePayService.prepay(request)`。
- 返回微信 Native 支付 `codeUrl`。

异常处理：

- 微信业务错误抛出 `WeChat Native Pay unified order failed: ...`
- 网络错误抛出 `WeChat Native Pay network error`
- 金额非法时抛出 `Order amount is invalid` 或 `Order amount must be greater than zero`

### 3. 新增 web 端支付接口

在 `OrderController` 新增：

- `POST /api/orders/{orderId}/pay/wechat-native`

接口行为：

- 使用 `Authorization` token 获取当前登录用户。
- 校验当前用户只能支付自己的订单。
- 校验订单状态必须为 `PENDING`。
- 调用 `wechatPayService.createNativeOrder(...)`。
- 返回微信 Native 支付二维码链接 `codeUrl`。

该接口和小程序端 `pay/wechat-miniapp` 的位置与调用方式保持一致，区别只在于：

- 小程序端返回 `wx.requestPayment` 参数。
- web 端返回 Native 支付二维码链接。

## 前端修改详情

### 1. 新增二维码生成依赖

在 `zhixi-website/frontend` 安装：

- `qrcode`

用途：

- 将后端返回的微信 Native `codeUrl` 转换为浏览器可展示的二维码图片 Data URL。

### 2. 新增 web 支付 API

在 `frontend/src/api.js` 新增：

- `payWechatNativeOrder(orderId)`
  - 调用 `POST /orders/{orderId}/pay/wechat-native`
  - 返回 `{ codeUrl }`

- `fetchOrderById(orderId)`
  - 调用 `GET /orders/{orderId}`
  - 用于支付后轮询订单状态

保留旧 `payOrder(orderId)` 方法未删除，避免影响其它旧调用方；但订单弹窗已经不再使用该 mock 支付接口。

### 3. 修改订单弹窗支付逻辑

在 `OrderModal.vue` 中：

- 引入 `qrcode`
- 引入 `payWechatNativeOrder`
- 引入 `fetchOrderById`
- 支付按钮从旧的 `payOrder` 改为调用 `payWechatNativeOrder`
- 收到 `codeUrl` 后生成二维码并展示
- 支付二维码生成后每 3 秒轮询一次订单详情
- 发现订单状态为 `PAID` 或 `PAID_PENDING_CASHBACK` 后：
  - 停止轮询
  - 展示支付成功提示
  - 触发 `success` 事件刷新用户侧订单/返现状态
- 组件卸载时清理轮询定时器

### 4. 新增二维码展示样式

在 `frontend/src/styles.css` 新增：

- `.wechat-pay-panel`
- `.wechat-pay-qr`
- `.wechat-pay-copy`

样式目标：

- 在订单弹窗内稳定展示二维码。
- 保持二维码固定尺寸，避免布局跳动。
- 显示扫码提示与支付状态同步文案。

## 当前支付流程

### web 端

1. 用户填写收货信息并提交订单。
2. 前端调用 `POST /api/orders` 创建订单。
3. 订单创建成功后展示微信支付按钮。
4. 用户点击微信支付。
5. 前端调用 `POST /api/orders/{orderId}/pay/wechat-native`。
6. 后端向微信支付 Native 下单，返回 `codeUrl`。
7. 前端用 `qrcode` 生成二维码。
8. 用户使用微信扫码支付。
9. 微信支付成功后回调 `/api/pay/wechat/notify`。
10. 后端验签并更新订单状态。
11. web 前端轮询 `GET /api/orders/{orderId}`。
12. 轮询到支付成功状态后刷新页面状态。

### 小程序端

小程序端仍保持原流程：

1. 调用 `POST /api/orders/{orderId}/pay/wechat-miniapp`
2. 后端返回 `timeStamp`、`nonceStr`、`package`、`signType`、`paySign`
3. 小程序调用 `wx.requestPayment`
4. 后端通过微信回调更新订单状态

## 注意事项

- web 端使用的是微信 Native 支付，需要商户号具备 Native 支付能力。
- 微信支付回调地址仍使用当前配置的 `WECHAT_PAY_NOTIFY_URL`。
- 支付成功依赖微信回调能正常到达线上后端。
- 本地开发环境若无法接收微信回调，二维码支付成功后本地页面可能无法自动轮询到已支付状态。
- 旧 `POST /api/orders/{orderId}/pay` mock 支付接口本次未删除，只是 web 订单弹窗不再调用它。

## 相关风险

- 如果商户平台未开通 Native 支付，`pay/wechat-native` 会返回微信下单失败。
- 如果 `app.wechat.pay.appid` 与商户平台 Native 支付产品配置不匹配，微信下单可能失败。
- 如果订单已不是 `PENDING` 状态，后端会拒绝再次生成支付二维码。
- 如果同一订单重复生成二维码，微信侧仍使用相同 `outTradeNo`，实际支付结果以微信订单状态和回调为准。
